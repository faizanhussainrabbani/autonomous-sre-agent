# Project Glossary

To ensure precise communication and prevent context drift, this glossary defines the canonical terms used across all SRE Agent documentation and codebase.

## 1. Event Types

*   **Alert:** A raw, unprocessed notification from an observability tool (e.g., "CPU > 90% for 5m"). Alerts are noisy and often grouped.
*   **Anomaly:** A statistically significant deviation from a learned baseline (as determined by the Detection Layer). An anomaly is an observation, not necessarily a problem.
*   **Incident:** A confirmed, actionable disruption to service health that requires diagnosis and remediation. An `Incident` is the canonical domain object (see `data_model.md`).

## 2. Actions & Outcomes

*   **Action:** The specific API call or Git commit proposed or executed by the agent (e.g., "Restart Pod X").
*   **Remediation:** The high-level intent or strategy to fix an incident (e.g., "Scale up the deployment to handle traffic"). Remediation is fulfilled by executing one or more Actions.
*   **Resolution:** The final state where the telemetry has returned to a healthy baseline *after* a Remediation has been applied.

## 3. Metrics & Scoring

*   **Confidence Score:** The internal probability (0.0 to 1.0) generated by the Intelligence Layer's RAG pipeline indicating how certain it is that its hypothesis is correct. Determines if the agent will act or escalate.
*   **Accuracy:** A historical, post-facto metric calculating what percentage of the agent's diagnoses matched the subsequent human-verified root cause. Used for Phase Graduation.
*   **Blast Radius:** 
    *   *As a Safety Limit:* The hard-coded maximum permissible scope of an autonomous action (e.g., "Max 20% of pods").
    *   *As an Assessment Metric:* The predicted impact area of an incident or an action, calculated via the Dependency Graph.

## 4. Phase States

*   **Shadow Mode (Phase 1):** The agent ingests telemetry, generates diagnoses, and logs its intended actions, but is strictly prohibited from executing them. Also referred to as "Observe mode" or "Read-only mode".
*   **Assist Mode (Phase 2):** The agent generates PRs or ChatOps prompts and awaits explicit human approval before executing actions.
*   **Autonomous Mode (Phase 3):** The agent executes actions immediately without human intervention for well-understood incident types.

## 5. Agent Behaviors & Architecture

*   **Canary Execution:** A safe deployment or remediation strategy where the agent applies a change to a small subset of instances first, monitoring for regressions before a full rollout.
*   **Fencing Token:** A monotonically increasing integer provided by the distributed lock manager. The agent passes this token with its K8s/Argo API calls to prevent stale writes if it experiences a GC pause or network partition.
*   **Hexagonal Architecture (Ports & Adapters):** The software design pattern used by the agent, isolating core business logic (`domain/`) from external systems (Prometheus, K8s) via interfaces (`ports/`) and concrete implementations (`adapters/`).
*   **MTTR (Mean Time To Recovery):** The average time taken from incident detection to final resolution. A primary KPI for the agent.
*   **Oscillation Loop:** A rapid, destructive cycle where the agent continually applies conflicting actions (e.g., scaling up then scaling down in quick succession). Prevented by the `cooling_off` period.
*   **RAG (Retrieval-Augmented Generation):** The pattern of retrieving relevant context (Post-Mortems, Runbooks) from a Vector DB and injecting it into the LLM prompt to ground its reasoning securely.
